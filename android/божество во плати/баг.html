Поиск багов Android-версии через ADB bugreport

Туториал для проекта Minecraft Legacy Remake: как правильно снять bugreport.zip, найти native crash, Java crash, ANR, ошибки Vulkan/SDL и понять, какой файл в C++ нужно исправлять.

Навигация

1. Цель 2. Подготовка 3. Снятие bugreport 4. Распаковка 5. Поиск ошибок 6. Классификация 7. Пример из проекта 8. Внутренние логи игры 9. Рабочий цикл 10. Чек-лист

1. Цель

bugreport.zip нужен, когда обычный logcat пустой, игра закрывается без сообщения, падает в native-коде или зависает после входа в мир.

Для твоего случая главный сценарий: запустить игру → зайти в мир → дождаться краша → сразу снять bugreport. Если снять bugreport до воспроизведения бага, нужных строк там может не быть.

2. Подготовка PowerShell

Задай переменные один раз в начале работы:

$adb = "C:\Users\Пользователь\AppData\Local\Android\Sdk\platform-tools\adb.exe"
$device = "PDNP05J000057115"
$folder = "C:\Users\Пользователь\Desktop\old-build\minecraft_legacy_2"
$bugreport = "$folder\bugreport.zip"
$out = "$folder\android_crash_logcat.txt"

Проверь, что телефон виден:

& $adb -s $device devices

Нормальный результат:

PDNP05J000057115    device
Если есть emulator-5554 offline, это не страшно. Просто всегда используй -s PDNP05J000057115.

3. Как правильно снять bugreport

1

Очисти старые логи

& $adb -s $device logcat -b all -c
2

Запусти игру

Можно вручную на телефоне или командой:

& $adb -s $device shell monkey -p com.minecraftlegacy.app -c android.intent.category.LAUNCHER 1
3

Повтори баг

На телефоне зайди в мир, дождись краша, зависания или неправильного поведения.

4

Сразу сними bugreport

& $adb -s $device bugreport "C:\Users\Пользователь\Desktop\old-build\minecraft_legacy_2\bugreport.zip"
5

Проверь файл

Get-Item $bugreport
Не делай так: logcat -c → bugreport без запуска игры. Тогда отчёт будет почти бесполезным.

4. Распаковка bugreport.zip

Распакуй отчёт в отдельную папку:

$extract = "$folder\bugreport_unpack"
Remove-Item $extract -Recurse -Force -ErrorAction SilentlyContinue
Expand-Archive $bugreport -DestinationPath $extract

Проверь, что появились файлы:

Get-ChildItem $extract -Recurse -File | Select-Object -First 30

5. Быстрый поиск ошибок

Главная команда поиска:

Get-ChildItem "$folder\bugreport_unpack" -Recurse -File |
  Select-String -Pattern "com.minecraftlegacy.app","libmain.so","SIGSEGV","SIGABRT","FATAL EXCEPTION","VK_ERROR","Vulkan","SDL","assert","ANR","tombstone" -ErrorAction SilentlyContinue

Если вывод слишком большой, сохрани результат в файл:

Get-ChildItem "$folder\bugreport_unpack" -Recurse -File |
  Select-String -Pattern "com.minecraftlegacy.app","libmain.so","SIGSEGV","SIGABRT","FATAL EXCEPTION","VK_ERROR","Vulkan","SDL","assert","ANR","tombstone" -ErrorAction SilentlyContinue |
  Out-File "$folder\bugreport_search_result.txt" -Encoding UTF8

Отдельные быстрые проверки:

# Native crash
Get-ChildItem $extract -Recurse -File | Select-String -Pattern "SIGSEGV","SIGABRT","libmain.so","tombstone" -ErrorAction SilentlyContinue

# Java crash
Get-ChildItem $extract -Recurse -File | Select-String -Pattern "FATAL EXCEPTION","AndroidRuntime","MainActivity" -ErrorAction SilentlyContinue

# Vulkan / SDL
Get-ChildItem $extract -Recurse -File | Select-String -Pattern "VK_ERROR","Vulkan","SDL_Vulkan","SDL" -ErrorAction SilentlyContinue

# Зависания
Get-ChildItem $extract -Recurse -File | Select-String -Pattern "ANR","Input dispatching timed out","not responding" -ErrorAction SilentlyContinue

6. Как понимать тип ошибки

Признак Тип бага Где искать в коде
SIGABRT, assertion failed Код сам аварийно завершил игру assert(...), abort(), std::terminate()
SIGSEGV Неправильное обращение к памяти C++ указатели, массивы, Vulkan/SDL handles, потоки
FATAL EXCEPTION Java/Kotlin crash MainActivity.java, AndroidManifest.xml, SDLActivity
ANR Игра зависла Главный поток, генерация чанков, бесконечный цикл, тяжёлая загрузка
VK_ERROR, VK_ERROR_DEVICE_LOST Проблема Vulkan/GPU renderer.cpp, создание surface, swapchain, buffer, memory
libmain.so Падение в C++ части игры src/: renderer, world streamer, generator, save
Если видишь libmain.so и рядом имя функции, это самое полезное место. Оно обычно показывает реальный C++ файл и цепочку вызовов.

7. Пример из твоего проекта

В одном из отчётов был найден такой тип ошибки:

Cmdline: com.minecraftlegacy.app
name: SDLThread
signal 6 (SIGABRT)
Abort message:
src/game/world_generator.cpp:1367:
run_mesh_builder_self_check(...):
assertion "seam_neighbor_mesh_faces == 5" failed

Цепочка:

run_mesh_builder_self_check()
ml::build_chunk_mesh(...)
ml::WorldStreamer::worker_loop()

Расшифровка

Это означает, что игра не упала случайно. Её остановил assert внутри проверки mesh builder-а. Такой баг не ловится обычным try/catch, потому что assert вызывает abort().

Что делать

// Было:
assert(seam_neighbor_mesh_faces == 5);

// Лучше для Android runtime:
if (seam_neighbor_mesh_faces != 5) {
    log_message(
        LogLevel::Warning,
        std::string("MeshBuilder self-check failed: seam_neighbor_mesh_faces=") +
        std::to_string(seam_neighbor_mesh_faces) +
        ", expected=5"
    );
    return;
}
Самопроверки можно оставлять для debug-тестов, но нельзя давать им убивать Android-игру в runtime.

8. Внутренние логи игры

Если установлен патч с логгером, попробуй найти файлы внутри приложения:

& $adb -s $device shell run-as com.minecraftlegacy.app find . -type f

Если увидишь latest.log или native_crash.log, вытащи их:

& $adb -s $device shell run-as com.minecraftlegacy.app cat ./files/logs/latest.log > "$folder\latest.log"
& $adb -s $device shell run-as com.minecraftlegacy.app cat ./files/logs/native_crash.log > "$folder\native_crash.log"
Если run-as пишет package not debuggable, значит APK собран не как debug. Переключи Build Variant на debug.

9. Рабочий цикл поиска багов

  1. Очистить логи: logcat -b all -c.
  2. Запустить игру.
  3. Повторить баг.
  4. Снять bugreport.zip.
  5. Распаковать отчёт.
  6. Найти SIGSEGV, SIGABRT, FATAL EXCEPTION, ANR, VK_ERROR.
  7. Определить файл и функцию.
  8. Исправить только это место.
  9. Пересобрать APK.
  10. Повторить тест.
Для каждого бага сохраняй отдельный архив: bugreport_01.zip, bugreport_02.zip, bugreport_after_fix.zip. Так проще сравнивать прогресс.
