# Minecraft Legacy Vulkan Project Plan

## 1. Цель проекта

Собрать кроссплатформенную voxel-игру в духе `Minecraft Legacy Console Edition` с максимально близким ощущением от игры, но на современном собственном стеке:

- `C++20`
- `Vulkan` как основной графический API
- целевые платформы: `Android`, `iOS`, `Windows`, `Linux`, `macOS`
- единый игровой мир между устройствами
- целевая сетевая модель: `P2P`
- поддержка будущих мини-игр на той же сетевой архитектуре

Первый обязательный результат: приложение запускается, создаёт мир по seed, генерирует чанки и показывает мир через свободную камеру.

## 2. Продуктовое направление

Приоритеты проекта:

1. Максимально близкий `legacy console edition` feel.
2. Один общий codebase для всех платформ.
3. Простая и расширяемая архитектура движка.
4. Быстрый вертикальный срез до состояния `мир + генерация + камера`.
5. Дальнейшее наращивание до игрока, блоков, сохранений и сетевой игры.

Что важно зафиксировать заранее:

- Текстуры сейчас не критичны.
- Разные типы блоков нужны уже на уровне архитектуры данных.
- На первом этапе можно использовать простые цвета или базовые материалы вместо полноценного texture atlas.
- Архитектура должна быть сразу пригодна для будущих:
  - разных блоков
  - инвентаря
  - разрушения и установки блоков
  - мира с сохранением
  - P2P-сессий
  - мини-игр

## 3. Технический стек

### Обязательная база

- Язык: `C++20`
- Сборка: `CMake` + `CMakePresets.json`
- Рендер: `Vulkan`
- Платформенный слой: `SDL3`
- macOS/iOS Vulkan-слой: `MoltenVK`
- Android toolchain: `Android NDK`
- Apple toolchain: `Xcode`

### Рекомендуемые библиотеки

- Аудио: `miniaudio`
- GPU texture pipeline: `KTX 2.0 / KTX-Software`
- Сеть: `ENet` как стартовая база для reliable UDP
- Графическая отладка: `RenderDoc`
- Профилирование: `Tracy` на следующем этапе

Примечание:

- По сетевому стеку нет 100% финального решения до проработки NAT traversal, host migration и mini-game flow.
- На старте достаточно проектировать интерфейсы так, чтобы позднее можно было заменить или надстроить transport-слой.

## 4. Архитектура проекта

Проект делится на 4 основных слоя:

### 4.1 platform

Отвечает за:

- окно и surface
- lifecycle приложения
- ввод
- gamepad
- файловую систему
- таймеры
- threading primitives
- сокеты и сетевые примитивы
- интеграцию с платформенными SDK

### 4.2 render

Отвечает за:

- инициализацию Vulkan
- device/swapchain/render pass pipeline
- загрузку и выгрузку GPU-ресурсов
- отрисовку chunk mesh
- материалы блоков
- будущую поддержку UI/hud render pass

### 4.3 game

Отвечает за:

- описание блоков
- мир
- чанки
- генерацию
- стриминг мира
- игрока
- физику/коллизию
- правила симуляции
- сохранение мира

### 4.4 net

Отвечает за:

- P2P-сессии
- репликацию мира и игроков
- синхронизацию событий
- host/client роли
- будущие мини-игры и их сетевые состояния

## 5. Ключевые игровые системы

### 5.1 Система блоков

Нужно сразу заложить расширяемую таблицу блоков:

- `air`
- `grass`
- `dirt`
- `stone`
- `water`

Позже будут добавлены:

- песок
- дерево
- листья
- стекло
- руды
- декоративные блоки
- блоки мини-игр и служебные интерактивные объекты

Для каждого блока в будущем нужны:

- `BlockId`
- флаги прозрачности
- флаги коллизии
- material/texture reference
- параметры разрушения
- параметры размещения

### 5.2 Мир и чанки

Минимальная модель данных:

- `BlockId`
- `Chunk`
- `ChunkCoord`
- `ChunkColumn`
- `World`
- `WorldStreamer`
- `WorldGenerator`

Требования:

- seed-based генерация
- детерминируемость генератора
- потоковая загрузка чанков вокруг камеры/игрока
- фоновая генерация и rebuild mesh
- система unload для дальних чанков

### 5.3 Генерация мира

Первый этап генерации:

- простая `heightmap`
- слои `grass/dirt/stone`
- базовая вода
- ровный рабочий baseline без сложных биомов

Позже:

- биомы
- пещеры
- руды
- деревья
- структуры
- legacy-подобные ограничения генерации

### 5.4 Камера и управление

Первый этап:

- свободная debug-камера
- движение по миру
- поворот камеры
- загрузка чанков по позиции камеры

Следующий этап:

- player controller
- коллизия
- прыжок
- гравитация
- выделение блока

## 6. Сетевое направление

Целевая модель:

- `P2P player-to-player`
- один мир доступен игрокам с разных устройств
- мини-игры строятся поверх той же сетевой сессии

Но в первый vertical slice сеть не входит как обязательная runtime-часть.

Что закладывается уже сейчас:

- разделение локального симуляционного состояния и сетевого представления
- сериализуемые структуры мира и событий
- понимание authority model
- возможность позднее реплицировать:
  - чанки
  - игроков
  - блоковые изменения
  - игровые события

Риски, которые надо будет закрыть отдельно:

- NAT traversal
- host migration
- рассинхрон мира
- читоустойчивость
- синхронизация мини-игр

## 7. Этапы разработки

### Этап 0. Инициализация проекта

Сделать:

- базовую структуру репозитория
- `CMake`-конфигурацию
- desktop-first запуск
- Vulkan bootstrap
- SDL3 интеграцию
- базовую систему логов
- заготовку platform/render/game/net модулей

Результат:

- проект собирается хотя бы на одной desktop-платформе
- есть окно и рабочий render loop

### Этап 1. Мир + генерация + камера

Сделать:

- описать блоки
- реализовать данные чанков
- написать первый генератор
- сделать chunk mesh generation
- стриминг чанков вокруг observer
- свободную камеру
- базовую отрисовку мира

Результат:

- запускается приложение
- виден voxel-мир
- мир строится по seed
- чанки догружаются вокруг камеры

Это главный ближайший milestone.

### Этап 2. Игрок и базовое взаимодействие

Сделать:

- player controller
- физику и коллизию
- выделение блока
- ломание блока
- постановку блока
- простую hotbar/inventory-заглушку

Результат:

- можно ходить по миру и менять блоки

### Этап 3. Сохранение мира

Сделать:

- формат сохранений
- сохранение изменённых чанков
- загрузку мира
- управление seed и world metadata

Результат:

- мир сохраняется и повторно открывается

### Этап 4. Базовый P2P multiplayer

Сделать:

- создание сессии
- подключение peer
- синхронизацию игроков
- синхронизацию изменений блоков
- синхронизацию чанков/области мира

Результат:

- несколько игроков входят в один мир

### Этап 5. Legacy-feel и контент

Сделать:

- gamepad-first UX
- меню и HUD
- более точное поведение движения
- расширение списка блоков
- текстуры и visual style
- аудио и ambience
- шлифовку feel под legacy

### Этап 6. Мини-игры

Сделать:

- режимы поверх общей сетевой архитектуры
- lobby/session flow
- правила раундов
- синхронизацию событий мини-игр

## 8. Ближайший рабочий план

Немедленный фокус:

1. Создать структуру исходников.
2. Поднять `SDL3 + Vulkan` desktop bootstrap.
3. Поднять базовые world data structures.
4. Реализовать простую генерацию чанков.
5. Реализовать meshing.
6. Подключить свободную камеру.
7. Вывести мир на экран.

До завершения этого этапа не распыляться на:

- сложный UI
- полноценные текстуры
- инвентарь
- продвинутую физику
- полноценный networking runtime

## 12. Критические риски

- Слишком ранняя попытка поддержать все платформы одновременно.
- Слишком раннее усложнение генерации мира.
- Попытка сделать точный legacy-feel до появления базового playability.
- Недооценка сложности `P2P` для мобильных сетей.
- Недооценка различий Vulkan runtime на desktop и `MoltenVK` на Apple-платформах.

Практический вывод:

- сначала нужен сильный desktop prototype
- затем Android
- затем Apple-платформы
- после этого уже стоит расширять сеть и legacy polish

## 13. Критерии успеха ближайшего milestone

Milestone считается завершённым, когда:

- приложение запускается
- Vulkan renderer стабильно рисует кадры
- есть seed мира
- есть генерация чанков
- мир виден на экране
- камера перемещается
- чанки стримятся вокруг камеры

## 14. Краткая памятка при возвращении в проект

Этот проект не про обычный Minecraft-клон в вакууме.

Это:

- `Minecraft Legacy Console Edition`-ориентированная игра
- на `C++20`
- на `Vulkan`
- с общим codebase под `Android/iOS/Windows/Linux/macOS`
- с будущим `P2P multiplayer`
- с будущими мини-играми

Текущий главный фокус:

- сначала добиться `world + generation + camera`
- не тратить время на текстуры и контент раньше времени
- архитектуру блоков сделать расширяемой сразу
- сеть проектировать заранее, но не реализовывать раньше базового vertical slice
