3. Краткая архитектурная характеристика

Тип архитектуры

Game loop architecture + modular monolith + engine/runtime split. Это не ECS: нет отдельного entity/component registry, основные роли распределены между Application, WorldStreamer, Renderer, PlatformApp, WorldGenerator, BlockRegistry и PlayerController.

Центр сборки

Application остаётся composition root: создаёт platform, renderer, block registry, player, save и world streamer. Минус: класс всё ещё перегружен меню, игровым циклом, debug HUD, взаимодействием с блоками и состояниями мира.

Самое важное изменение

Light subsystem выделен отдельно. Теперь lighting — это не просто цвет вершин, а отдельный runtime stage между генерацией чанка и построением меша.

Горячие зоны

CPU: генерация, освещение, greedy meshing, player collision. GPU/driver: upload буферов чанков, draw calls по секциям, occlusion/cave/frustum culling. IO: сохранение dirty чанков.

5. Build system и зависимости

CMake

Проект собирается как C++20. На desktop создаётся minecraft_legacy, на Android — shared library main. SDL3 подтягивается через FetchContent, Vulkan ищется через SDK или Android library.

Shaders

GLSL лежит в assets/textures/texture_pack/classic/shaders. CMake вызывает glslc и генерирует *.spv. Поэтому *.spv — generated output, а не главный исходник.

Windows

build.bat и build_release.bat запускают CMake presets. После сборки CMake копирует assets и shaders рядом с exe.

Android

android/app/build.gradle использует externalNativeBuild CMake, compileSdk 36, minSdk 29, ndkVersion 30.0.14904198, arm64-v8a.

Diagnostics

В ветке release нет встроенного CSV PerfRecorder. Для проверки оптимизаций уже доступны DebugHudData и StreamingStats: очереди чанков, last_generate_ms, last_light_ms, last_mesh_ms, upload/culling counters.

Build pipeline

8. Platform, Input и Audio

PlatformApp

Слой SDL3: окно, fullscreen, Vulkan surface, keyboard/mouse/gamepad events, mouse capture, audio stream, UI sounds, music.

InputState

Единая структура ввода, чтобы Application, PlayerController и DebugCamera не зависели напрямую от SDL events.

Audio

WAV используется для UI-событий, OGG — для фоновой музыки через STB Vorbis. На Android нужны отдельные asset-пути и иногда другой способ чтения.

Граница ответственности

platform не должен знать про блоки, чанки и правила игры. Он только отдаёт состояние ввода и базовые platform services.

10. World data model

Мир хранится чанками 16 x 384 x 16: kWorldMinY = -64, kWorldMaxY = 319. Высота 384 блока разделена на 24 секции по 16 блоков, что позволяет renderer и culling работать не только целыми чанками, но и вертикальными секциями.

13. Light subsystem — проверено в v10

Смысл: calculate_chunk_light() строит sky light не только по центру чанка, а в padded-области с границей kLightBorder = 4. Для устранения "тёмных углов" теперь используются не только 4 стороны (W/E/N/S), но и 4 диагональных угла (NW/NE/SW/SE).
Ограничение текущей реализации: по коду основной расчёт сейчас направлен на Sky Light. ChunkLight.block_light и light_emission уже есть в модели, но полноценное распространение света от факелов/лавы как отдельный BFS-слой лучше вынести в следующий этап, чтобы не обещать больше, чем реально делает текущая ветка.

15. Chunk state machine

Актуальная state machine стала более подробной. Это полезно для защиты, потому что можно показать, где именно чанк ожидает соседей и почему сначала может появиться временный меш.

Состояния чанка

stateDiagram-v2 [*] --> Requested Requested --> TerrainGenerated: GenerateTerrain complete TerrainGenerated --> Decorated: Decorate complete Decorated --> WaitingForNeighbors: light provisional Decorated --> LightReady: light final enough WaitingForNeighbors --> LightReady: borders ready / relight LightReady --> MeshQueued: BuildMesh(provisional) or BuildMesh(final) queued MeshQueued --> MeshBuilt: worker complete MeshBuilt --> UploadQueued: pending upload UploadQueued --> UploadedToGPU: drained by Application UploadedToGPU --> Visible: visible_chunks Visible --> MeshQueued: block changed / leaves mode / final edge fix Visible --> [*]: outside radius unload
Примечание после сверки: в enum действительно присутствуют промежуточные статусы VerticalSkyReady, LightPropagating, EdgeCheckPending. Этот блок надо защищать как архитектурную карту жизненного цикла чанка: часть стадий сейчас может быть подготовлена в enum/дизайне, но не обязана существовать как отдельный полностью завершённый runtime-проход.

16. Provisional mesh и Final mesh внутри BuildMesh

В ветке release нет отдельных enum job-типов BuildMesh(provisional) и BuildMesh(final). Фактически есть один job type BuildMesh, а различие fast/final выражается через ChunkMeshSnapshot::provisional и debug-счётчики queued_fast_meshes/queued_final_meshes. Это важно для защиты: fast/final — правильная архитектурная идея, но в коде она реализована как режим одного mesh job.

Provisional BuildMesh

Создаётся, когда соседние light borders ещё не полностью готовы. Игрок быстрее видит чанк, но на стыках может быть временная неточность света.

Final BuildMesh

Создаётся после готовности света/границ. Такой mesh должен заменить provisional upload и убрать резкие тёмные края.

Замена provisional на final через один BuildMesh

flowchart TD Light["CalculateLight"] --> Provisional{"light provisional?"} Provisional -->|yes| MeshFast["BuildMesh with snapshot.provisional = true"] MeshFast --> UploadFast["upload provisional mesh"] UploadFast --> Borders{"light borders ready?"} Borders -->|no| Wait["wait / relight neighbors"] Wait --> Borders Borders -->|yes| MeshFinal["BuildMesh with snapshot.provisional = false"] Provisional -->|no| MeshFinal MeshFinal --> UploadFinal["upload final mesh"] UploadFinal --> Replace["replace stale/provisional upload"]

17. Border lighting fix

Твой баг с резкой темнотой на границе чанка относится именно сюда. Когда чанк строит face на краю, ему нужны не только свои блоки, но и соседние блоки/свет. Без этого один чанк считает край “открытым”, другой “закрытым”, и появляется ломаный свет.

Rebuild соседейqueue_light_self_and_neighbors... и queue_rebuild_self_and_neighbors...Пересчитываются ли соседние чанки после изменения блока у края?

18. Greedy meshing

Greedy meshing объединяет соседние faces с одинаковыми свойствами. После обновления ключевым условием становится не только одинаковый блок/текстура/render type, но и совместимость lighting key, иначе нельзя объединять лица с разной освещённостью.

Решение по face

flowchart TD Block["current BlockId"] --> Type{"render type"} Type -->|Opaque| Opaque{"neighbor hides face?"} Type -->|Cutout leaves| Cutout{"fast or fancy leaves?"} Type -->|Transparent water| Water{"same water neighbor?"} Opaque -->|hidden| Skip["skip face"] Opaque -->|visible| LightKey["sample light + AO"] Cutout --> LightKey Water --> LightKey LightKey --> Mask["put FaceKey into greedy mask"] Mask --> Merge["merge same texture direction light"] Merge --> Section["append to opaque/cutout/transparent section"]

19. GPU upload budget — проверено в v10

Чтобы генерация чанков не создавала резкие просадки FPS, Application забирает задачи на загрузку через drain_pending_uploads_by_budget. WorldStreamer сортирует их по приоритету (дистанция + конус взгляда) и отдает только тот объем, который вписывается в лимиты байт и количества секций на кадр.

Upload pipeline с троттлингом

flowchart TD Pending["pending_uploads vector"] --> Sort["Sort by priority observer position + forward"] Sort --> Budget["drain_pending_uploads_by_budget byte/section limit"] Budget --> Validate["check expected rebuild serial"] Validate -->|stale| Drop["record stale upload drop"] Validate -->|valid| Renderer["renderer.upload_chunk_mesh"] Renderer --> Buffers["create/reuse GPU buffers per section"] Buffers --> Ready["chunk ready for draw calls"]

20. Culling и отрисовка лишних полигонов

Проект уже имеет несколько уровней отсечения: frustum culling, section culling, cave/surface culling и экспериментальный occlusion culling. Самая важная идея для оптимизации — скрывать не только целые чанки, но и вертикальные секции, а ещё лучше — связные области, если они гарантированно не видны.

УровеньЧто отсекаетЧто улучшить дальше
Hidden faces в meshingВнутренние грани между блокамиСтрого учитывать прозрачность/листья/воду
Frustum cullingСекции вне камерыПроверить AABB секций, а не только чанков
Cave/surface cullingПещеры, когда игрок на поверхности, и поверхность, когда игрок под крышейДобавить portal/connected visibility
Occlusion cullingСекции закрытые ближней геометриейСделать hierarchical Z или software occluder grid стабильнее
Greedy meshСокращает количество faces/draw dataНе ломать lighting/AO при объединении

21. Cave visibility — проверено в v10

Система определяет, находится ли игрок под “крышей”, через Application::update_cave_visibility_frame(): она сэмплирует несколько колонн над камерой, считает непрозрачные roof_blocks, применяет hysteresis и передаёт в renderer структуру Renderer::CaveVisibilityFrame. Это позволяет отсекать поверхностные секции в пещерах и наоборот.

CaveVisibilityFrame

cave_mode: активен ли режим пещеры.
roof_blocks: количество непрозрачных блоков над камерой.
camera_chunk_x/z: текущий чанк для точной привязки отсечения.

Логика отсечения

Секция считается Cave, если она ниже kSeaLevel и закрыта сверху. Секция считается Surface, если она выше или открыта небу. В cave_mode Surface-секции отбрасываются.

28. План доработок “как у команды”

  1. Снять базовый perf-профиль. Пройти один и тот же маршрут по миру с включённым Debug HUD, записать FPS/очереди/upload/light/mesh значения вручную или добавить отдельный CSV-профилировщик перед крупными оптимизациями.

  6. Оптимизировать GPU uploads. Пул буферов, лимит байт в кадр, приоритет по камере, профилирование upload_ms.
  7. Добавить тестовые миры. Плоская крыша на границе чанков, туннель через границы, вода/листья на границе, высокая гора, пещера под поверхностью.
  8. Документировать каждый публичный сценарий. Где изменить блок, где свет пересчитывается, где mesh обновляется, где renderer получает upload.

31. Итог для защиты

Формулировка: проект сейчас — кроссплатформенный C++20/Vulkan/SDL3 voxel runtime с модульным монолитом. Главный поток идёт через Application, мир грузится через WorldStreamer, чанки проходят стадии генерации, декорации, отдельного расчёта освещения, fast/final mesh rebuild и GPU upload. Главные новые архитектурные точки — вынесенный Light subsystem . Light subsystem должен убрать резкие перепады света на границах чанков за счёт border snapshots, readiness states и финального rebuild, а отдельный CSV-профилировщик стоит добавить следующим этапом, чтобы доказывать оптимизацию через CSV-метрики, а не только визуально.

Коротко, что сказать преподавателю

  • Документ обновлён под текущую ветку release, а не под старый вариант.
  • Освещение вынесено в отдельный модуль src/game/light.
  • Chunk pipeline теперь двухфазный по mesh: быстрый provisional и финальный corrected.
  • WorldStreamer — самая нагруженная система; её надо разделять при следующем рефакторинге.
  • Сначала надо использовать Debug HUD/StreamingStats как baseline, а CSV-профилировщик добавить как отдельный инструмент для замеров.
  • Для оптимизации отрисовки уже есть секции чанков и culling metadata, дальше нужен более строгий cave/occlusion visibility.
